## 2. Unify Generation Path in server.go

### Problem

Three functions perform server config generation with ~80% identical code:
- `GenerateServer()` (lines 49-124) — single-server, no cross-server state
- `generateServerWithState()` (lines 560-653) — multi-server with state, returns dirty map
- `regenerateServer()` (lines 655-744) — nearly identical to `generateServerWithState()` but without dirty tracking, and processes certificates after marshaling (different order)

Additionally, `regenerateServer()` has a subtle ordering issue: certificates are resolved
after `marshalOptions()`, meaning the pin-sha256 stored in state during certificate resolution
may not be available to outbounds that were already built.

### Solution

**Approach C: single generation path.**

1. `generateServerWithState()` becomes the sole function for building server configs.
   Certificates are always resolved before outbounds (correct order for pin-sha256).
2. `GenerateServer()` becomes a thin wrapper:
   ```go
   func GenerateServer(dir string, cfg config.Config, genCfg GenerateConfig) (GenerateResult, error) {
       state := NewServerState()
       result, _, err := generateServerWithState(dir, filepath.Base(dir), cfg, genCfg, state)
       return result, err
   }
   ```
3. `regenerateServer()` is deleted entirely.
4. `generateWithDAG()` calls `generateServerWithState()` directly for dirty servers.
   The second call to `provisionCrossServerUsers()` is a no-op (users already exist),
   and the dirty map from regeneration is empty.
5. `buildOutboundsWithState()` replaces `buildOutbounds()` (unused after refactor).
   `buildOutbounds()` is deleted.

### Files Changed

- `generate/server.go` — remove `GenerateServer()` body, `regenerateServer()`,
  `buildOutbounds()`
- `generate/server_test.go` — update tests that call removed functions

## 3. Structured Error Attribution in validate

### Problem

`checkOutboundInboundRefs()` returns `[]error`. `runPhase1()` uses fragile string
matching to attribute errors to servers:
```go
if strings.Contains(ce.Error(), fmt.Sprintf("server %q ", name)) {
```
The trailing space is a hack to distinguish source server ("server X outbound")
from target server ("on server Y"). This breaks if error message formats change.

Additionally, `_ = graph` on line 129 discards the graph that was just built
for cycle detection (the cycle error is returned above, so the graph variable
is truly unused after that point).

### Solution

1. `checkOutboundInboundRefs()` returns `map[string][]error` (server name → errors)
   instead of `[]error`. The function iterates over configs and buckets errors by
   the originating server.
2. `runPhase1()` directly maps errors from the returned map instead of string matching.
3. Remove `_ = graph` on line 129 (graph is already validated above, variable is unused).

### Files Changed

- `validate/check.go` — change `checkOutboundInboundRefs()` signature and `runPhase1()` logic
- `validate/check_test.go` — update tests for new error structure

## 4. Replace Panics with Errors in Crypto Code

### Problem

Seven `panic()` calls in `generate/credentials.go` and `generate/certs.go` violate
the project convention of returning errors. While crypto/rand failures are
catastrophic, panics prevent graceful error reporting in CLI output and make
testing error paths impossible.

### Solution

Change all crypto generation functions to return errors:

| Function | Current Signature | New Signature |
|----------|------------------|---------------|
| `GenerateUUID()` | `() string` | `() (string, error)` |
| `GeneratePassword()` | `() string` | `() (string, error)` |
| `GenerateX25519KeyPair()` | `() (string, string)` | `() (string, string, error)` |
| `GenerateShortID()` | `() string` | `() (string, error)` |
| `GenerateSelfSignedCert()` | `(string) ([]byte, ed25519.PrivateKey)` | `(string) ([]byte, ed25519.PrivateKey, error)` |
| `GenerateSelfSignedCertPEM()` | `(string) ([]byte, []byte)` | `(string) ([]byte, []byte, error)` |

Error wrapping convention:
```go
return "", fmt.Errorf("generate uuid: %w", err)
```

### Caller Updates

- `generateUserCreds()` — returns `(UserCreds, error)` instead of `UserCreds`
- `resolveRealityKeys()` — propagates key generation errors up through creds
- `resolveObfsPassword()` — propagates password generation errors
- `resolveCredentials()` — returns `error` as second return value
- `generateServerWithState()` — wraps credential errors with context
- `resolveCertificates()` — wraps cert generation errors with context

### Files Changed

- `generate/credentials.go` — all functions return error
- `generate/certs.go` — all functions return error
- `generate/server.go` — callers propagate errors
- `generate/credentials_test.go` — test error returns
- `generate/certs_test.go` — test error returns
- `generate/server_test.go` — update for new error signatures
- `generate/inbound_test.go` — update for new error signatures

## 5. Validate ListenPort Range

### Problem

`config.Inbound.ListenPort` is `int` with no validation. Values outside 0-65535
pass silently. `intToUint32()` in `generate/inbound.go` returns 0 for negative
values but overflows for values > MaxUint32.

### Solution

Add validation in `config.Validate()`:
```go
if in.ListenPort < 0 || in.ListenPort > 65535 {
    errs = append(errs, fmt.Errorf(
        "inbound %q: listen_port %d is out of range (0-65535)", in.Tag, in.ListenPort,
    ))
}
```

`ListenPort == 0` is valid (field not specified, used for tun inbounds).

### Files Changed

- `config/load.go` — add port range check in `Validate()`
- `config/load_test.go` — add test cases for invalid ports
